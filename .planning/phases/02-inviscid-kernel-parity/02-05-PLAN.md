**In AirfoilAnalysisService.cs**, modify the `AnalyzeInviscid` method (line ~57-70) to check `settings.InviscidSolverType` before dispatching:

```csharp
public InviscidAnalysisResult AnalyzeInviscid(
    AirfoilGeometry geometry,
    double angleOfAttackDegrees,
    AnalysisSettings? settings = null)
{
    if (geometry is null)
    {
        throw new ArgumentNullException(nameof(geometry));
    }

    settings ??= new AnalysisSettings();

    if (settings.InviscidSolverType == InviscidSolverType.LinearVortex)
    {
        return AnalyzeInviscidLinearVortex(geometry, angleOfAttackDegrees, settings);
    }

    var preparedSystem = PrepareInviscidSystem(geometry, settings);
    return inviscidSolver.Analyze(preparedSystem, angleOfAttackDegrees, settings.FreestreamVelocity, settings.MachNumber);
}
```

Add a private method `AnalyzeInviscidLinearVortex` that:
1. Extracts coordinates from `geometry.Points` into `double[]` arrays.
2. Calls `LinearVortexInviscidSolver.AnalyzeInviscid(x, y, count, angleOfAttackDegrees, settings.PanelCount, settings.MachNumber)`.
3. Adapts the `LinearVortexInviscidResult` to `InviscidAnalysisResult` by:
   - Also running `PrepareInviscidSystem` to get the `PanelMesh` (needed by downstream consumers and the return type).
   - Mapping fields: `LiftCoefficient`, `MomentCoefficient` (-> `MomentCoefficientQuarterChord`), `PressureDragCoefficient` (-> `DragCoefficient` and `CorrectedPressureIntegratedDragCoefficient`).
   - Using `LiftCoefficient` for both `CorrectedPressureIntegratedLiftCoefficient` and `PressureIntegratedLiftCoefficient`.
   - Setting `Circulation` to `0.0` (not computed by linear-vortex path; a placeholder -- the linear-vortex solver does not produce a Kutta circulation value in the same form).
   - Setting `SourceStrengths` to `Array.Empty<double>()` and `VortexStrength` to `0.0` (these are Hess-Smith panel strengths, not applicable to linear-vortex).
   - Setting `PressureSamples` to `Array.Empty<PressureCoefficientSample>()` (Cp is available in `LinearVortexInviscidResult.PressureCoefficients` but the `PressureCoefficientSample` type requires `AirfoilPoint` + `TangentialVelocity` which are not straightforward to map; leave empty for now -- Phase 3 viscous coupling will need to revisit this adapter).
   - Creating a `WakeGeometry` with empty points: `new WakeGeometry(Array.Empty<WakePoint>())`.
   - Using `angleOfAttackDegrees` for the `AngleOfAttackDegrees` field, `settings.MachNumber` for `MachNumber`.

The `PrepareInviscidSystem` call to get the mesh is not strictly necessary for the linear-vortex path since the mesh is only used by downstream consumers that read `result.Mesh`. For now, generate the mesh anyway to keep the return type complete. This adds ~0.5ms overhead. If profiling shows concern later, this can be lazily evaluated.

IMPORTANT: Do NOT modify any other public methods (`SweepInviscidAlpha`, `SweepDisplacementCoupledAlpha`, etc.). Those still use `this.inviscidSolver` (HessSmith). The linear-vortex path through sweep methods will be wired in Phase 3 when the viscous coupling is ready. This gap closure ONLY needs to wire the single `AnalyzeInviscid` method to prove selectability.
