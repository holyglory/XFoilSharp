**Defect 2 -- Add DUE/DDS forced-change terms to VDEL RHS:**

In ViscousNewtonAssembler.BuildNewtonSystem, at the VDEL assignment block (around line 262-267):

1. Before the VDEL loop, compute the DUE and DDS values for the two-station interval. These are the edge velocity and displacement thickness changes between the current and previous station:
   ```csharp
   // DUE/DDS: edge velocity and displacement thickness changes
   // Fortran: DUE1 = 0, DDS1 = 0 (previous station delta = 0 for RHS)
   //          DUE2 = 0, DDS2 = 0 (current station delta = 0 for RHS)
   // BUT: The forced-change terms use the actual Ue and D* perturbations
   // from the Newton update of the PREVIOUS iteration. For the first
   // iteration or when no prior update exists, these are zero.
   // In Fortran SETBL, DUE and DDS come from the UEDG/DSTR changes
   // since the last SETBL call.
   ```

   Actually, examining the Fortran more carefully: In SETBL, DUE1/DUE2/DDS1/DDS2 are the differences between the current Ue/D* and the values from the previous global iteration. For the VDEL RHS, these represent the "forced change" that happened outside the Newton system (i.e., from the edge velocity update via DIJ).

   The implementation should track the previous-iteration values of UEDG and DSTR. Before the side/station loop, save the current UEDG and DSTR. Then compute:
   ```csharp
   double due1 = uei_prev1 - uei_previter_1; // Change in Ue at station 1 since last SETBL
   double dds1 = dsi_prev1 - dsi_previter_1; // Change in D* at station 1 since last SETBL
   double due2 = uei - uei_previter;          // Change in Ue at station 2 since last SETBL
   double dds2 = dsi - dsi_previter;          // Change in D* at station 2 since last SETBL
   ```

   **Simpler approach matching Fortran exactly:** The DUE/DDS terms in Fortran SETBL represent the change in Ue and D* from the last time SETBL was called. Add parameters `double[,] uedgPrev` and `double[,] dstrPrev` to BuildNewtonSystem (or pass them as the "previous iteration" arrays). The ViscousSolverEngine maintains these by saving UEDG/DSTR before each call to BuildNewtonSystem.

   In the station march loop, compute:
   ```csharp
   // Before the for loop over stations, also track previous-station due/dds:
   double due1 = 0, dds1 = 0; // Initialized for first station

   // Inside the loop, before VDEL assignment:
   double due2 = uei - uedgPrev[ibl, side];
   double dds2 = dsi - dstrPrev[ibl, side];
   // where dstrPrev is computed from massPrev/uedgPrev
   ```

   Then modify the VDEL assignment:
   ```csharp
   for (int k = 0; k < 3; k++)
   {
       vdel[k, 0, iv] = localResult.Residual[k]
                       + localResult.VS1[k, 3] * due1
                       + localResult.VS1[k, 4] * dds1
                       + localResult.VS2[k, 3] * due2
                       + localResult.VS2[k, 4] * dds2;
       vdel[k, 1, iv] = 0.0;
   }
   ```

   **IMPORTANT Fortran index mapping:** In the Fortran code, VS1/VS2 column indices are 1-based. `VS1(k,4)` maps to C# `VS1[k,3]` and `VS1(k,3)` maps to C# `VS1[k,2]`. However, looking at the existing C# code for the VA/VB blocks (lines 255-259), the C# code already uses `VS2[k,0]` for Ctau/ampl and `VS2[k,1]` for Theta. The VS column layout in C# is:
   - Column 0: Ctau/Amplification
   - Column 1: Theta
   - Column 2: D* (mass defect)
   - Column 3: Ue (edge velocity)
   - Column 4: x (arc length)

   So the Fortran `VS1(k,4)*DUE` (Fortran 1-based col 4 = Ue sensitivity) maps to C# `VS1[k,3]*due` and `VS1(k,3)*DDS` (Fortran col 3 = D* sensitivity) maps to C# `VS1[k,2]*dds`.

   **Corrected VDEL assignment:**
   ```csharp
   for (int k = 0; k < 3; k++)
   {
       vdel[k, 0, iv] = localResult.Residual[k]
                       + localResult.VS1[k, 3] * due1  // VS1 Ue sensitivity * DUE1
                       + localResult.VS1[k, 2] * dds1  // VS1 D* sensitivity * DDS1
                       + localResult.VS2[k, 3] * due2  // VS2 Ue sensitivity * DUE2
                       + localResult.VS2[k, 2] * dds2; // VS2 D* sensitivity * DDS2
       vdel[k, 1, iv] = 0.0;
   }
   ```

   Update `due1 = due2; dds1 = dds2;` at the end of each station iteration (for the next station's "previous" values).

2. Add `double[,] uedgPrev, double[,] massPrev` parameters to `BuildNewtonSystem`. The caller (ViscousSolverEngine) will maintain these arrays.

3. In ViscousSolverEngine, before calling BuildNewtonSystem, save copies of blState.UEDG and blState.MASS as the "previous iteration" arrays. On the first iteration, initialize them to the current values (so DUE/DDS = 0 on the first call, matching Fortran behavior).

Restructure the Newton coupling iteration loop in ViscousSolverEngine.SolveViscousFromInviscid (lines ~140-287) to match Fortran VISCAL order.

**Current (incorrect) order:**
```
a. MarchBoundaryLayer(...)        // Pre-modifies theta/dstar/ctau
b. UpdateEdgeVelocityDIJCoupling  // DIJ coupling
c. BuildNewtonSystem (SETBL)      // Assembles around wrong state
d. BlockTridiagonalSolver.Solve   // Solves wrong system
e. ApplyNewtonUpdate (conditional)// Rarely helps
f. STMOVE
g. CL/CD/CM
```

**Correct Fortran VISCAL order:**
```
a. BuildNewtonSystem (SETBL)      // Assemble BL system at current state
b. BlockTridiagonalSolver.Solve   // Solve the block-tridiagonal system
c. ApplyNewtonUpdate (UPDATE)     // Always apply with RLXBL relaxation
d. Edge velocity update via DIJ   // Update Ue from displacement thickness changes (QVFUE)
e. STMOVE                         // Relocate stagnation point
f. CL/CD/CM                       // Track convergence
```

Specific changes:

1. **Remove MarchBoundaryLayer from the primary loop.** The BL march was used as a fallback because Newton diverged. With correct Jacobians (defects 1+2 fixed), Newton should converge on its own. Keep MarchBoundaryLayer as a method in the class (it may be useful for initialization) but do NOT call it inside the `for (int iter = ...)` loop.

2. **Reorder the loop body:**
   ```csharp
   for (int iter = 0; iter < maxIter; iter++)
   {
       // Save previous-iteration UEDG/MASS for DUE/DDS computation
       Array.Copy(blState.UEDG, uedgPrev, blState.UEDG.Length);
       Array.Copy(blState.MASS, massPrev, blState.MASS.Length);

       // a. SETBL: Build Newton system
       double rmsbl = ViscousNewtonAssembler.BuildNewtonSystem(
           blState, newtonSystem, dij, settings,
           isAlphaPrescribed: true, wakeGap,
           tkbl, qinfbl, tkbl_ms,
           hstinv, hstinv_ms,
           rstbl, rstbl_ms,
           reybl, reybl_re, reybl_ms, HvRat,
           isp, n, debugWriter,
           uedgPrev, massPrev);  // New params for DUE/DDS

       // b. BLSOLV: Solve block-tridiagonal system
       BlockTridiagonalSolver.Solve(newtonSystem, vaccel: 0.01, debugWriter: debugWriter);

       // c. UPDATE: Apply Newton update (always, with relaxation)
       var (rlx, updatedRms, newTrustRadius, accepted) =
           ViscousNewtonUpdater.ApplyNewtonUpdate(
               blState, newtonSystem, settings.ViscousSolverMode,
               hstinv, wakeGap, trustRadius, rmsbl, rmsbl,
               dij, isp, n, debugWriter);
       trustRadius = newTrustRadius;

       // d. QVFUE: Edge velocity update via DIJ
       UpdateEdgeVelocityDIJCoupling(blState, ueInv, dij, isp, n, nWake, rlxDij);

       // e. STMOVE: Relocate stagnation point
       // ... (existing STMOVE code, unchanged)

       // f. CL/CD/CM and convergence check
       // ... (use rmsbl for convergence, not marchRms)
   }
   ```

3. **Fix convergence check.** Use `rmsbl` (the Newton RMS from BuildNewtonSystem) for convergence checking instead of `marchRms`. The Fortran VISCAL checks RMSBL < VATEFM (tolerance). Convergence is declared when `rmsbl < tolerance`.

4. **Remove the "save-try-revert" pattern** for Newton updates. With correct Jacobians, Newton updates should always be applied (with relaxation from RLXBL in the updater). Remove the `postNewtonRms < marchRms * 0.99` guard. The ViscousNewtonUpdater already has its own relaxation logic -- trust it.

5. **Adaptive DIJ relaxation:** Keep the adaptive rlxDij logic but key it off `rmsbl` instead of `marchRms`:
   ```csharp
   if (rmsbl < prevRmsbl) rlxDij = Math.Min(rlxDij * 1.1, 0.5);
   else rlxDij = Math.Max(rlxDij * 0.7, 0.05);
   prevRmsbl = rmsbl;
   ```

6. **Initialize uedgPrev and massPrev** before the loop:
   ```csharp
   double[,] uedgPrev = (double[,])blState.UEDG.Clone();
   double[,] massPrev = (double[,])blState.MASS.Clone();
   ```
   This ensures DUE/DDS = 0 on the first iteration (matching Fortran behavior).

7. **Update convergence history** to log `rmsbl` instead of `marchRms`.

8. **Update debug logging** to match the reordered steps (POST_UPDATE after UPDATE, not after BL march).
