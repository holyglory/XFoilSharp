<interfaces>
From BoundaryLayerSystemAssembler.cs (the VS1/VS2/VSREZ output that feeds into VA/VB/VM/VDEL):
The assembler produces a local 4x5 system (VS1[4,5], VS2[4,5], VSREZ[4], VSR[4], VSM[4], VSX[4])
which SETBL (ViscousNewtonAssembler) stuffs into the global Newton system blocks.

The key chain is:
1. BoundaryLayerCorrelations.cs computes BL correlations (Cf, Hk, Hs, etc.) with Jacobian sensitivities
2. BoundaryLayerSystemAssembler.AssembleLocalSystem calls BLSYS/BLDIF/TRDIF to build VS1/VS2/VSREZ
3. ViscousNewtonAssembler.BuildNewtonSystem stuffs VS1/VS2/VSREZ into global VA/VB/VM/VDEL arrays
4. BlockTridiagonalSolver.Solve performs Gaussian elimination on VA/VB/VM to get VDEL solutions
5. ViscousNewtonUpdater.ApplyNewtonUpdate reads VDEL solutions and applies Newton deltas

The divergence could be at any of steps 2-5. The comparison script will narrow it down.

Two solver modes exist (per CONTEXT.md locked decisions):
- ViscousSolverMode.TrustRegion (DEFAULT): max 50 iters, tol 1e-6, save/restore/trust-radius logic
- ViscousSolverMode.XFoilRelaxation: max 20 iters, tol 1e-4, RLXBL adaptive under-relaxation
Both modes share the same assembly code (SETBL/BLSOLV/UPDATE). Fixes to assembly correctness benefit both modes.

   - Context: the BL_STATE at that station (both Fortran and C# values)
   - The previous station's values (to see if divergence propagated or originated here)

5. Also report a summary table:
   - Per-iteration: RMSBL comparison, CL/CD comparison
   - First divergence point per category (BL_STATE, VA, VB, VDEL, VSREZ)

6. Report RMSBL convergence trajectory: for each iteration in both dumps, print RMSBL and whether it decreased from previous iteration. At the end, print a clear verdict: "RMSBL CONVERGING" (if monotonically decreasing for 3+ consecutive iterations) or "RMSBL DIVERGING" (if increasing).

**CRITICAL: Dual-mode architecture (per CONTEXT.md locked decisions)**

The codebase has two solver modes (ViscousSolverMode enum):
- **TrustRegion (DEFAULT):** max 50 iterations, tolerance 1e-6, save/restore state + trust-radius shrink on divergence. This is the PRODUCTION default and MUST be preserved.
- **XFoilRelaxation:** max 20 iterations, tolerance 1e-4, RLXBL adaptive under-relaxation (DHI=1.5, DLO=-0.5). This is the XFoil-compatible verification path.

Both modes share the same assembly code (BuildNewtonSystem, BlockTridiagonalSolver.Solve, ApplyNewtonUpdate). The Jacobian fixes in this task improve BOTH modes because the assembly chain is shared. The divergence is in the shared assembly -- not in the mode-specific update logic.

The goal of this task is:
1. Fix the shared assembly bugs (BoundaryLayerSystemAssembler, ViscousNewtonAssembler, BlockTridiagonalSolver) so that Newton corrections are correct.

**General debugging methodology:**
1. Start from the first divergence point identified by the comparison script.
2. Open the C# source file and the corresponding Fortran source side by side.
3. Compare the specific computation at the divergent station line-by-line.
4. Common issues to look for:
   - **Sign errors** in Jacobian sensitivities (D2_U2 = -DSI/UEI vs +DSI/UEI)
   - **Missing terms** in VM column assembly (the mass defect sensitivity chain VS1(k,3)*D1_M(JV) + VS1(k,4)*U1_M(JV) etc.)
   - **Wrong indexing** in U2_M computation: U2_M(JV) = -VTI(IBL,IS)*VTI(JBL,JS)*DIJ(I,J) -- check that VTI signs, panel index I, and system line JV are all correct
   - **Missing DUE/DDS forced change terms** in VDEL residual
   - **Missing XI sensitivity terms** (XI_ULE1*ULE1_M, XI_ULE2*ULE2_M) in VM column
   - **TESYS vs BLSYS** selection at TE-to-wake junction (IBL == IBLTE+1)
   - **VZ block** assembly for TE coupling (only assembled once when IS=1, IV=IVTE1)
   - **COM1/COM2 state transfer** between stations (the "previous station" state must be properly carried forward)
   - **BLSOLV VA block structure**: Fortran uses VA(eq, col, iv) where col=1,2 not col=1,3. The third "column" is handled through the VM mass defect matrix. Check C# array dimensions match.
   - **BLSOLV VACCEL threshold**: The VM column elimination uses VACCEL thresholds. Check that the C# implementation applies the same thresholds.

5. After each fix, re-run the diagnostic dump test and the comparison script to verify the fix narrows the divergence.
